## Milestone Definition of Done (from ROADMAP overview)

> "v1.1 Batch Feedback Pipeline replaces the current 1-edit-per-deploy feedback flow with a batched submission model: client stages multiple edits in a single session via a corner chip + panel UI, submits them as one POST, the server creates one GitHub issue containing all edits, and the Claude Code Action produces one branch / one commit / one PR / one deploy. Targets a ~10× reduction in deploys per client review pass."

Two phases:
1. **Phase 4: Batch Pipeline Implementation** — everything that ships in the PR (client UI, server v2 schema, issue construction, Action manual, cache-bust, editor-flow fence)
2. **Phase 5: Post-Deploy Verification** — canary checks that only run after Vercel deploys the merged code

The milestone is shippable when Phase 4 lands (a single ~450 LOC PR) AND Phase 5 canaries confirm the deployed pipeline works end-to-end against real GitHub + Claude Action infrastructure.

## Tech Debt (deferred — non-blocking)

### Phase 4 follow-ups

1. **D-03 Vercel tier reconciliation.** Both `MAX_BATCH_BYTES` constants are 3 MB (Hobby-tier-safe). Before merge, owner should visit `vercel.com/MSizzle/moulin-a-reves/settings/general` and confirm tier. If Pro+ with body-size override available, both constants can be lifted to 30 MB in the same PR (also bumping `FEEDBACK_INJECT_VER` to `'4'`).

2. **Smoke harness retarget seam (Phase 5 dependency).** `scripts/smoke-feedback-v2.mjs` is currently unit-only (imports `submit.ts`, stubs GitHub fetch). Phase 5 planning must choose: (a) add `TARGET_URL` env-var mode to the harness (real `fetch()` when set), (b) write a separate `curl`-based canary script, or (c) other. Recommended: (a), so the same 5 scenarios cover both unit and integration modes.

3. **WR-01 (INFO):** `MAX_IMAGE_BYTES` (12 MB per-photo) > `MAX_BATCH_BYTES` (3 MB per-batch). First photo over 3 MB trips "batch is full" message instead of "this photo is too large." Low UX priority; consider distinguishing the messages in a follow-up.

4. **WR-05 (INFO):** `MAX_IMAGE_BYTES` imported in `submit.ts` but missing from the `void` ritual (lines 16–21). Cosmetic.

5. **SUMMARY frontmatter hygiene.** 10 of 11 Phase 4 plan SUMMARY files use non-standard `requirements:` key or omit `requirements-completed:` entirely. Only `04-04-SUMMARY.md` uses the standard `requirements-completed: [...]` array. Truth-of-record is VERIFICATION.md (unambiguous); this is metadata hygiene for future audit-mileage. Consider a one-time backfill or a SUMMARY template update.
