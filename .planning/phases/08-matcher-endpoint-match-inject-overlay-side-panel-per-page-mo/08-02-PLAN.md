<objective>
Ship the auth-gated `POST /api/feedback/match` endpoint. The endpoint:
1. Short-circuits with 401 if the `maison_session` cookie fails `checkAuth()` (D-09).
2. Parses `{ route, editList }` from the JSON body; rejects malformed body with 400 and over-cap input with 422 (MATCH-07).
3. Loads `/edit-catalogs/<route>.json` via **same-origin HTTP fetch** against `new URL('/edit-catalogs/' + slug + '.json', request.url)`; returns 404 if missing (MATCH-02). HTTP fetch — NOT filesystem read — because Vercel Function `cwd` is the function bundle root (not the repo root), so the catalog static assets shipped to `dist/client/edit-catalogs/` are not reachable via `process.cwd()`. The same-origin fetch path works because catalogs ship as public static assets per Phase 7 CATALOG-06 and Astro's Vercel adapter serves `dist/client/*` at the deployment root.
4. Tokenizes `editList` per MATCH-05 (bullets `- * 1. 2.`, paragraphs, section headers as context-not-lines).
5. Calls Claude Haiku 4.5 via `@anthropic-ai/sdk` with **forced tool-use** for structured output (D-10, D-11, D-12, D-13).
6. Validates every returned `primaryId` + `alternates[]` ID against the catalog's in-memory `Set<id>`; strips unknown IDs; sets `primaryId = null` when validation fails (D-14, MATCH-04).
7. Returns `{ ok: true, matchSetId: 'ms_<uuid>', buildSha: <catalog.buildSha>, matches: [...] }` on success.
8. Returns `{ ok: false, error: 'matcher_unavailable' }` 500 if `ANTHROPIC_API_KEY` is missing OR the retried Anthropic call fails (MATCH-06, D-13).

Add `@anthropic-ai/sdk` to `dependencies` (NOT `devDependencies` — runtime use).

Purpose: This is the spine of the per-page review flow. The inject (Plan 03) and the panel (Plans 04+05+06) both depend on the response shape locked here. The OPS-02 fence is maintained passively (no edits to fenced files).

Output: One new server route file (~250 lines TypeScript), one `package.json` + `package-lock.json` update with the SDK added.

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Client → Vercel Function `match.ts` | Untrusted JSON body (`route`, `editList`) crosses here over HTTPS; cookie auth check is first gate |
| Function → Anthropic API (claude-haiku-4-5) | Outbound: prompt contains user's `editList` (untrusted text); server-side `tool_choice` forces structured response |
| Function → same-origin static catalog asset (`/edit-catalogs/<slug>.json`) | Same-origin HTTP fetch from the auth-gated function host back to its OWN deployment's public static asset tree. The origin is derived from `request.url`, NOT a user-controlled hostname, so the fetch cannot be redirected to an attacker-controlled host. Asset is public per CATALOG-06. |
| Server env → process | `ANTHROPIC_API_KEY` is server-only; OPS-03 / MATCH-06 prevent leakage to client bundle |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-08-02-01 | Spoofing | `match.ts` POST without valid `maison_session` cookie | mitigate | First line of POST handler: `if (!(await checkAuth(request))) return unauthorized();` — short-circuits BEFORE any work (catalog fetch, Anthropic call). Identical pattern to `submit.ts`, `validate.ts`, `clarify.ts`, `status/[issueNumber].ts`. |
| T-08-02-02 | Tampering | Client-supplied `editList` injection into Haiku prompt | mitigate | (a) `tool_choice: { type: 'tool', name: 'match_edits' }` forces structured tool-use output — Haiku CANNOT bypass the schema (D-11). (b) Server-side validation pass: every returned `primaryId` and every `alternates[]` entry is checked against `Set<catalogId>` built from the loaded catalog; unknown ids are silently stripped (D-14). Even a fully-cooperative malicious prompt cannot smuggle an unknown ID into the response. |
| T-08-02-03 | Tampering | Path-traversal via crafted `route` body field | mitigate | `route` validation requires non-empty string starting with `/` AND rejects strings containing `..` or null bytes BEFORE slug derivation. `routeToSlug` strips slashes and produces a relative path component that becomes the URL path of a same-origin GET. Even a crafted slug cannot escape the deployment's static catalog tree because the URL is built from `new URL('/edit-catalogs/' + slug + '.json', request.url)` — `URL` parsing normalizes path traversal characters at the URL layer (any `../` in `slug` is resolved relative to `/edit-catalogs/`, never escaping the deployment origin). `fetch` returning non-2xx → caller returns 404. |
| T-08-02-04 | Information Disclosure | `ANTHROPIC_API_KEY` leaked to client | mitigate | (a) Var read only via `import.meta.env.ANTHROPIC_API_KEY` in this server file. (b) Task 3 grep gate asserts zero references in `public/` AND zero references in `dist/client/*` after build. (c) Operator action (Vercel env Production scope per OPS-03) keeps the key out of preview / dev deployments unless explicitly added. |
| T-08-02-05 | Information Disclosure | Catalog content exposed via 404 path or error message | accept | Catalogs ship to production (CATALOG-06) — they ARE public. A 404 reveals only "this route has no catalog", which is the intended UX signal. Error messages do not echo back `route` to attacker-controlled output beyond the route they already supplied. |
| T-08-02-06 | Denial of Service | Cost / token blowup via large `editList` or `catalog` | mitigate | MATCH-07 caps: `editList` ≤ 10000 chars (422 over), catalog ≤ 150 entries (422 over). `max_tokens` capped at `min(8192, 64 + 96*lineCount)` (D-12). Single one-shot Anthropic call per request — no chunking. 25s timeout via `AbortController` (D-13). One retry on transient 5xx/429. |
| T-08-02-07 | Denial of Service | Unauthenticated flood of POST requests | mitigate | Auth gate short-circuits BEFORE any work. Cookie sessions are HMAC-signed and 30-day TTL. Vercel platform-level rate limiting applies as defense-in-depth. (No per-user rate limit implemented at the app layer — consistent with `submit.ts` design.) |
| T-08-02-08 | Repudiation | Operator denies they submitted a match request | accept | No persistent logging of match requests is implemented. The Anthropic call is logged on Anthropic's side (request ID in their console). Match-set IDs are generated server-side (`ms_<uuid>`) but only stashed in client-side sessionStorage. Repudiation is low-risk because the matcher does NOT create persistent state (no issues, no PRs); the v1.1 submit pipeline (Plan 05's Approve write-through) creates the auditable artifact. |
| T-08-02-09 | Elevation of Privilege | Catalog file outside the deployment's `/edit-catalogs/` tree fetched | mitigate | T-08-02-03 mitigations apply at the URL layer. Additionally: the same-origin fetch goes through Vercel's normal static-asset router — there is no filesystem path lookup, no `eval`, no dynamic require. Any request to a path outside `/edit-catalogs/<slug>.json` simply 404s at the Vercel edge. |
| T-08-02-10 | Tampering | Stale match set replay via crafted client sessionStorage | mitigate | Matcher returns `buildSha` in the response. The inject (Plan 03 / OVERLAY-02 / D-15..D-17) compares `matchSet.buildSha` against the deployed `<meta name="x-build-sha">`; mismatch → no pins painted, banner shown, user must re-run match. Server-side, the matcher itself does NOT check session continuity (a stale match set could be replayed against the matcher, but doing so produces a fresh response with the current buildSha — no harm). |
| T-08-02-11 | Spoofing | Same-origin catalog fetch hijacked to a different deployment | mitigate | The catalog fetch URL is constructed via `new URL('/edit-catalogs/' + slug + '.json', request.url)`. `request.url` is set by the Vercel runtime from the incoming request and is NOT controllable by the body. The fetch therefore always targets the same Vercel deployment that received the POST — production POSTs hit production catalogs, preview POSTs hit preview catalogs. There is no way for an attacker to redirect this fetch to a different host. (Defense-in-depth on T-08-02-03.) |
