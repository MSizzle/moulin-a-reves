Per CONTEXT D-05 and D-07, the assertion-and-cleanup body MUST use a try/finally pattern. The try block performs these steps in order:

Step 1: POST the payload via canaryPost. Assert response.status === 200. On failure log a FAIL line including the actual status code and the response body text, then throw.

Step 2: Parse the response body as JSON. Assert body.ok === true and typeof body.issueNumber === 'number' and typeof body.issueUrl === 'string'. On failure log a FAIL line with the stringified body, then throw.

Step 3: Log a PASS line containing the string `PASS: v1 canary HTTP 200 with issueNumber=` followed by the integer value.

Step 4: Capture body.issueNumber to an outer-scope variable named issueNumber (the finally block needs it for cleanup; Task 2 reads it for the evidence file).

Step 5: Invoke execFileSync from node:child_process with the gh command and the arguments ['issue', 'view', String(issueNumber), '--json', 'title,labels,body'] with encoding utf8. Parse the resulting JSON.

Step 6: Assert the parsed title starts with the literal string `[TEST] [Feedback] change wording — `. On failure log a FAIL line with the actual title and throw.

Step 7: Assert the parsed labels array contains an entry where name equals the literal string `client-feedback-test`. On failure log a FAIL line with the labels array and throw.

Step 8: Assert the parsed body does NOT contain the substring `[Feedback] batch of`. This is a negative grep — confirms the v1 dispatch arm in submit.ts handled the request, not v2. On failure log a FAIL line including the first 200 chars of body and throw.

Step 9: Log a PASS line `PASS: v1 canary issue body matches single-edit shape`.

The finally block (CRITICAL — must run even on assertion failure, per CONTEXT D-07):

If issueNumber was captured, invoke execFileSync with gh and arguments ['issue', 'comment', String(issueNumber), '--body', 'Closed by canary — Phase 5 OPS-04 verification (' + new Date().toISOString() + ')']. Wrap in try/catch. Then invoke execFileSync with gh and arguments ['issue', 'close', String(issueNumber)]. Wrap in try/catch. If either gh call throws, log a WARN line `WARN: cleanup partial — issue <N> may need manual close` with the underlying error message, but ONLY change the process exit code to 3 if the try block had succeeded (use an outer flag assertionsPassed set to true at the end of the try block); if assertions had already failed, keep the assertion-failure exit code (1).

Exit semantics per CONTEXT discretion: exit 0 on PASS (all assertions hold and cleanup succeeded); exit 1 on assertion failure; exit 2 on setup pre-check failure; exit 3 if assertions PASS but cleanup fails.

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-05-06 | Spoofing | maison_session cookie | mitigate | createSession() HMAC-signs with DASHBOARD_PASSWORD. Spoofed cookies fail checkAuth (returns 401). The canary uses a fresh real cookie per invocation; no replay risk. |
| T-05-07 | Tampering | Test issue label namespace pollution | mitigate | Test issue uses client-feedback-test label (Action skips). Cleanup in finally closes the issue with a "Closed by canary" comment for audit trail. The label namespace is unaffected. |
| T-05-08 | Repudiation | Lost evidence of canary run | mitigate | Evidence file is committed to git in Task 2; the canary run is dated (frontmatter + log timestamps + git commit). Re-running the canary creates a new file or appends a new section. |
| T-05-09 | Information disclosure | Canary payload contains client-identifiable data | accept | Payload content is fixed in source (Canary test edit — Phase 5 verification + ISO timestamp); no real client data, no internal URLs, no PII. Public issue body shows only the canary marker text. |
| T-05-10 | Denial of service | Repeated canary runs spamming issues | accept | OPS-04 canary is opt-in (`npm run canary:v1`); each run creates 1 issue and closes it immediately. Even worst-case operator-error of running 100 times produces 100 closed test issues — non-blocking and trivially archivable. |
| T-05-11 | Elevation of privilege | gh CLI scoped beyond intent | mitigate | The canary uses only gh issue view, gh issue comment, gh issue close — all read-or-write-to-own-issue operations within the existing gh auth scope. No org-wide writes, no PR merges, no repo settings touched. |
| T-05-12 | Spoofing | TARGET_URL pointing somewhere malicious | mitigate | DEPLOY_URL defaults to the production URL in canary.sh. Override is operator-explicit (DEPLOY_URL=...); operator-level threat model. Documented in canary.sh header. |
