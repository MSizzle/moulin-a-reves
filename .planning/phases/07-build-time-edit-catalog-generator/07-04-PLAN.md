<objective>
Build a Node-side index of all string values in `src/content/**/*.md` frontmatter so the catalog walker (07-03) can classify hardcoded DOM text into one of two `kind: hardcoded-text` sub-states: rows where the text is anchored to a content-collection source (catalog can express a stable locator pointing at the .md file's frontmatter field) versus rows where the text is hardcoded inline in a .astro file (catalog emits `requiresManualSelection: true` so the matcher in Phase 8 cannot return an ambiguous locator).

Purpose: CATALOG-04. The Phase 8 matcher endpoint trusts the catalog. If the catalog promises a stable locator for a hardcoded-text row, the matcher will return that ID confidently and Phase 8 PANEL Approve will stash a v2 staged edit whose `domPath`-only locator fails the v1.1 `signalCount() >= 2` rule (validate.ts:54). Failing fast at catalog-emission time by flagging `requiresManualSelection: true` keeps the v1.1 validator the single source of truth without bypass.

Output: A new module `src/integrations/edit-catalog/content-index.mjs` exporting `buildContentIndex()` and `normalizeText()`, plus a node:test unit test pinning the contract. The walker (07-03) imports the index and queries it per text node.
</objective>

The index needs to capture string-valued frontmatter fields (title, tagline, summary, eyebrow, amenities[], gallery[].alt, gallery[].caption, etc.) and discard numbers, booleans, slugs, and image paths. Heuristic: capture any string value whose normalized length is >= 8 characters AND whose value does NOT match `^/images/` or `^https?://`.

Index entry shape:
  Map<normalizedText, Array<{ file: string, fieldPath: string }>>

  normalizedText = text.replace(/\s+/g, ' ').trim()
  file = repo-relative path like 'src/content/homes/le-moulin.md'
  fieldPath = dot-notation like 'tagline', 'amenities[0]', 'gallery[2].alt'

A value can appear in multiple files (e.g. the title "Le Moulin" appears in both homes/le-moulin.md and pages/le-moulin.md). The Map value is an Array so the walker (07-03) can decide what to do with collisions (current decision: if a normalized text matches >= 1 content source, emit `requiresManualSelection: false` and pick the first file/field; the matcher endpoint in Phase 8 will see the ambiguity downstream — out of scope here).

gray-matter is already in dependencies (used by Astro pages today — see src/pages/index.astro:7-8). No new dep needed.
